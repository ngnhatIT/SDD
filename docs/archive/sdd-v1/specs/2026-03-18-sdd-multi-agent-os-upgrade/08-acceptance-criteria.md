# Acceptance Criteria

| AC ID | Requirement | Acceptance Criteria |
| --- | --- | --- |
| `AC-01` | `REQ-01` | The canonical live path still preserves repository authority order, source-of-truth rules, anti-hallucination discipline, schema binding, and ask-before-break handling. |
| `AC-02` | `REQ-02` | There is one canonical execution routing surface and one short execution contract per major task family that explicitly lists inputs, reads, outputs, forbidden assumptions, and stop conditions. |
| `AC-03` | `REQ-03`, `REQ-04` | Codex, Claude, and GitHub Copilot each have explicit operating profiles, and prompt files are thin intent adapters instead of duplicated workflow manuals. |
| `AC-04` | `REQ-05` | Spec authoring is documented as a governed workflow from raw inputs through reviewable feature-package outputs, with a dedicated authoring checklist. |
| `AC-05` | `REQ-06` | Repeated fragile rules are owned by reusable enforcement checklists rather than restated in multiple prompts and stage docs. |
| `AC-06` | `REQ-07` | Main-path entrypoints clearly distinguish canonical, support, compatibility, and historical material, and they do not route agents through migration or cleanup history for routine work. |
| `AC-07` | `REQ-08` | Retained compatibility bridges are short, explicitly non-canonical, and point back to the live execution or governance source rather than carrying a second workflow body. |
