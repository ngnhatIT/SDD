# Failure Modes

| Failure Mode | Why Dangerous | Quick Detection | Corrective Action |
| --- | --- | --- | --- |
| skipped reinforcement when required | non-trivial work proceeds with no grounded ambiguity or risk record | `reinforcement.md` is missing, stale, or too thin to defend the next step | stop, write or refresh `reinforcement.md`, then continue from the grounded scope |
| implementation not aligned with `spec_pack.md` | changed behavior, files, or claims drift outside approved scope | diff or `verification.md` cannot map work back to the pack or acceptance criteria | narrow the change back to the pack, or update `spec_pack.md` before continuing |
| coded without reading the spec | implementation choices become guesswork and traceability collapses | code or review notes cannot cite the originating `spec_pack.md` requirements or acceptance targets | stop, reload `spec_pack.md`, restate the governed scope, and only then continue |
| placeholder artifact passes locally | weak governance evidence gives a false sense of completion | artifact body is empty, template-like, or dominated by `TODO`, `pending`, `none`, or placeholders | fail validation, replace placeholders with grounded content, and rerun validation |
| mixed unrelated tasks in one folder | evidence becomes ambiguous and downstream review or audit cannot tell which lifecycle is authoritative | one folder contains multiple unrelated scopes, or review and audit do not point back to the same `spec_pack.md` | split the unrelated scope into a new governed folder and keep one primary lifecycle per folder |
| review with weak evidence | `review.md` becomes opinionated commentary instead of grounded findings | findings cannot be traced to inspected code, files, standards, or the originating spec | reopen the review, list scope and files inspected, then rewrite findings against evidence |
| audit that mutates code | audit loses independence and hides implementation under an inspection label | `git status` shows changes or the audit cannot honestly say `Code Modified: no` | revert audit-only edits, reroute to `fix` or `implement` if changes are needed, and keep `audit.md` findings-only |
| closeout without the required artifact | work ends in chat or code only and cannot be verified later | `verification.md`, `review.md`, or `audit.md` is missing or the validator fails | update the correct artifact, rerun the validator, and do not close the task until the folder is consistent |
