# Acceptance Criteria

| ID | Requirement Link | Acceptance Criterion | Evidence |
| --- | --- | --- | --- |
| `AC-01` | `REQ-01` | `A canonical minimal-sufficient-context policy exists under governance and operationally defines baseline reads, risk-based deepening, execution-brief substitution, lightweight-path allowance, full-path loading triggers, and anti-hallucination rules for reduced context.` | New governance policy plus updated context or prompt cross-references |
| `AC-02` | `REQ-02` | `Definition of Done clearly states the completion bar for tiny-fix, reduced-path fix, full-path governed implementation, review-only, docs-only, audit-only, hotfix, and recovery or resume work without creating duplicate completion authorities.` | Updated `definition-of-done.md` and aligned supporting checklist or routing references |
| `AC-03` | `REQ-03` | `Lightweight paths explicitly state when they are allowed, the minimum artifacts required, the mandatory evidence or report, and when they must escalate to reduced-path or full-path handling.` | Updated task-mode routing and operator guidance |
| `AC-04` | `REQ-04` | `A practical framework-health-metrics document exists under ai-ops, defines each signal, explains why it matters, shows how to record it lightly, and avoids inventing a heavy analytics system.` | New metrics document plus ai-ops discovery update |
| `AC-05` | `REQ-05` | `Operators can discover the new policy, completion rules, lightweight paths, and metrics from existing canonical entrypoints without adding a competing process hub.` | Minimal updates to readmes, loading order, task profiles, and prompt guides |
