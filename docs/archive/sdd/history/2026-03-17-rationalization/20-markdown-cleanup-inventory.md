# Markdown Cleanup Inventory

## 1. Audit Objective

Inventory and classify every repository markdown file before any cleanup action, with special focus on distinguishing active SDD framework files from execution artifacts, historical residue, compatibility aliases, and ad hoc outputs.

Audit date: `2026-03-17`

## 2. Audit Method

1. Loaded the repository operating contract and mandatory SDD context in the order required by `AGENTS.md` and `docs/sdd/context/ai-loading-order.md`.
2. Enumerated every `.md` file present before this audit report was added (`406` pre-existing files total).
3. Collected per-file metadata: relative path, heading or title-derived purpose, directory role, and exact repo text references where present.
4. Read the framework-defining files plus suspicious outliers manually: migration/history docs, cleanup reports, spec-pack outputs, validator fixtures, review-only folders, and compatibility template aliases.
5. Classified each file conservatively using explicit repository evidence. Where provenance or downstream dependency remained unclear, the file was kept out of delete scope and marked `ARCHIVE` or `NEEDS-HUMAN-CONFIRMATION`.

## 3. Classification Rules

- `KEEP`: canonical framework file, governed feature artifact, explicit example, or validator fixture with ongoing operational value.
- `KEEP-TIGHTEN`: still useful and retained, but non-authoritative, generated, or drift-prone enough that it should stay out of the core framework path or be monitored for future tightening.
- `ARCHIVE`: historical, review-only, or cleanup-trace material that still has audit value but does not belong in the active framework surface.
- `DELETE-CANDIDATE`: high-confidence removable file with no canonical role, no meaningful references, and no evident traceability obligation. No files reached this confidence threshold in this audit.
- `NEEDS-HUMAN-CONFIRMATION`: looks removable or relocatable, but explicit repository notes, compatibility concerns, or unclear provenance make silent cleanup unsafe.

Interpretation rules used in this report:

- `Canonical = Yes` means the file is explicitly part of the authoritative SDD framework or ADR layer.
- `In Active SDD Process = Yes` means the file belongs to the normal governed delivery flow; `Support` means retained helper/example/fixture; `History` means traceability only.
- `Generated Result File = Yes` means the file is a generated spec-pack/brief or a run-specific audit/report/plan artifact, not a human-authored canonical source of truth.
- `Referenced` reports exact repo text references when found; `Pattern` means the file is operationally required by framework convention even if not directly linked by exact full path.
